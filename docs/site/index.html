Skills — marketplace map

Personal Claude marketplace. A growing collection of focused, single-purpose plugins for keeping code navigable as it grows. Three layers: a marketplace (the container) holds plugins (topical families) which hold skills (the verbs you actually invoke).

How to read this: scroll for a 30-second overview, or jump to any section from the left. Every domain graph is interactive — click a node for its details, drag to rearrange.

Built on Ousterhout's deep-module principles. Self-contained skills (no shared imports). Language-agnostic (universal core + per-stack heuristics).

Install — one source tree, five agents

Two channels: Claude Code and agy import the plugins natively (namespaced /<plugin>:<verb>); Codex, opencode, and Cursor auto-discover the flat mirror .agents/skills/. Shortcut for any harness — tell your agent: “Fetch and follow instructions from https://raw.githubusercontent.com/ChenPaulYu/skills/main/INSTALL.md”. Or by hand, in Claude Code:

/plugin marketplace add ChenPaulYu/skills
/plugin install nav@skills        # + shape@skills · research@skills · frame@skills · reflect@skills

In Antigravity CLI (agy) — native Claude Code plugin import, same namespace, no conversion. Clone the repo, then:

agy plugin install plugins/nav    # + plugins/shape · plugins/research · plugins/frame · plugins/reflect

Codex, opencode, and Cursor need no install step inside this repo — all three auto-discover the generated flat mirror .agents/skills/ (nav:audit → nav-audit; no plugin namespace). For global use, run node scripts/build-codex.mjs --sync-global --profile build --dedupe-global-roots: it creates ~/.agents/skills/, installs compiled flat skills there, and syncs only the needed runtime artifacts into ~/.codex/. See README → Codex compatibility.

Two more routes: the skills.sh CLI (npx skills add ChenPaulYu/skills — pick the prefixed skill set) and, for Cursor, the native plugin form (each plugin ships a .cursor-plugin/plugin.json; symlink into ~/.cursor/plugins/local/). Full guides: README → Install.

Iterating on this marketplace itself? Use a local absolute path instead of the GitHub handle — see README → Local development.

Architecture — three layers, one container

The marketplace registers via .claude-plugin/marketplace.json and lists which plugins live inside. Each plugin declares itself via .claude-plugin/plugin.json and exposes skills under skills/<verb>/SKILL.md. Slash commands surface as /<plugin>:<skill>.

plugin nav anatomy — assess, transform, describe, plan, execute

Eight skills, one verb each — audit is the read-only source of truth that plan (and sync's grounding pass) inlines.
  • Source · audit — assess against the 8 rules (read-only).
  • Move · refactor — verbatim move + test-gate, behaviour-preserving (heavy-touch, narrow scope).
  • Add · do — execute a small behaviour-changing change, disciplined, no artifact; refactor's twin. The inject↔check sub-agent bracket promoted to a standalone verb (ADR-023).
  • Headers · sync — file-top headers (per-file navigability; continuous, per-change).
  • Map · map — the repo codebase map (per-repo navigability; periodic, batched). Re-split from sync by cadence (ADR-029, superseding the ADR-019 merge).
  • Docs · compose — author/restructure a prose document as a deep module; sync's prose sibling (sync = the code-file header, compose = the prose body). ADR-049.
  • Meta · plan — the lone orchestrator: grounds a spec → clarifies → writes a plan artifact you review before executing.
  • Tour · tour — conversational walkthrough of what/how/why, ending in a mandatory correction loop against the agent's own model; consumes map, never invokes it. ADR-082.
The meta-skill (plan) doesn't auto-act on what it surfaces — it writes a plan artifact the user reads before executing. It inlines audit's protocol but follows the "scan transcript first" preamble — if audit ran this session, it reuses that output instead of re-running grep (ADR-006). audit itself is read-only; to act on its findings, run /nav:sync (gated) or /nav:refactor. For a small, already-decided behaviour change, /nav:do executes it directly — same deep-module/header discipline, no plan artifact.

plugin shape anatomy — converge · project · reconcile · build

Skills grouped by verb around a blueprints/ hub.
  • converge · mockup (render-to-decide) · elicit (grounded grill → a principle) · survey (map a decision space, report the diff vs your understanding; summoned or offered by elicit; ADR-074) · probe (design + run a minimal experiment when a fork can't be settled by argument; summoned or the destination of frame:dialectic's guarded offer; ADR-075) · dogfood (use the built feature for real → friction + coverage gaps; borrows /verify·/run, ADR-033) · position (canon-authoring campaign → core docs; graduation-grown, mirror of reconcile; ADR-035).
  • project · align — decide now/next/later WITH the user → plan.md + regenerate overview.html (the mirror of /nav:sync's map).
  • reconcile · amend a stale fact in place, prune/consolidate a wholly-stale doc, or retire a shipped-decision mockup (salvage → prune, git = deep archive; ADR-037); write-gated. A decision-change → /shape:elicit.
  • build · setup — scaffold a NEW project to a verified baseline (archetype references + stack principles; done = chain green; ADR-036) · build — drive In-progress to done — per item: ground via /nav:plan → implement → verify (screenshot vs mockup) → ship; autonomous but confidence-gated; parallel only by an approved schedule (ADR-040).
Forward-motion half (nav = maintenance half). Three seams, one-way (shape → nav): blueprints/ hands off to /nav:plan; reconcile reads /nav:sync; build orchestrates /nav:plan + /nav:refactor per item.

plugin research anatomy — dissect · untangle · critique

Three of research's four skills shown here (provenance lives in the DOMAINS card above) — dissect gives the nodes; untangle gives the edges (how the documents relate); critique assesses one argument document adversarially into a referee report.
  • dissect · decompose one or more documents into Gap / Claim / Mechanism / Evidence / Conclusion. Dispatches a sub-agent per document to read it in full (inject → read → check, ADR-008 discipline); main agent checks the returned note before saving. Single doc or batch — one note per document. Output saved to notes/.
  • untangle · the relational structure across N sources — lineage (who builds on whom) / clusters (parallel approaches) / contradictions / contested ground. Reuses dissect notes from notes/. Claim is optional: with it, adds a positioning layer (where your claim sits, the gap, related-work priority). Output: one note in notes/.
  • critique · adversarially assess one argument document into a referee report — claim↔evidence audit (validated ≠ claimed) + self-attack every finding before it ships (cut non-survivors → cut-log). Two coupled cores: ground the attack surfaces to recompute-depth, then refute. Consumes dissect; the rating stays with the reviewer. Output: analysis note + review draft + cut-log.
Standalone — no nav/shape dependency. Candidate skill remaining: feedback (design implications from competitor empirical results). See ADR-027, ADR-030, ADR-031.
The unit of analysis is the argument, not the document. Gap/Claim/Mechanism/Evidence makes cross-document comparison possible — lay multiple dissection notes side by side and the prior-art landscape becomes visible. Inject your own claim to get the "Implications" section: what does this document leave open that yours addresses?
