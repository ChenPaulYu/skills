# ADR 096 — Relay closure semantics: every Discussion is closed by its initiator

**Status**: accepted. **Amended by [ADR-097](docs/adr/097-relay-receipt-default.md)**: the "checks the reaction list, then closes" description in the FYI bullet below described a manual poll because at the time no non-`fyi` Announcement carried a receipt obligation at all — mentioning someone was pure notice-tier traffic. ADR-097 changes that: a non-`fyi` Announcement is now receipt-default, and its author's closure is machine-nudged (`SETTLE close-announcement` once every named recipient has reacted) rather than polled by hand. The manual-poll description below now applies only to the narrower `fyi`-labeled case, which ADR-097 keeps exactly as this ADR specified it.
**Date**: 2026-07-22
**Refines**: [ADR-092](docs/adr/092-relay-native-lifecycle-completion.md), [ADR-093](docs/adr/093-relay-obligations-vs-notices.md)

## Context

Relay's lifecycle matrix (`plugins/relay/CLAUDE.md`, ADR-092) states a native completion signal for every object category, but it never named who is authorized to take the *closure* action itself for two categories: FYI Announcements and non-Q&A Discussions (Ideas, or any thread opened to converge something without GitHub's native accepted-answer affordance). Q&A already had an answer — the Discussion author accepts an answer, then closes — but nothing generalized that rule, and FYI had no closure owner at all beyond "no closure obligation."

The gap was latent until a downstream consumer that treats an object's raw open/closed state as itself meaningful — open = outstanding, closed = clean, the shape of a conformance-style dashboard — made the ambiguity visible: closing an FYI or an Ideas Discussion changed what that dashboard reported, so *who* is allowed to flip that bit, and what the flip means, stopped being cosmetic.

## Decision

**Every Discussion is closed by its initiator.** One closure-owner class, no split by object category or by how many people the object addresses:

- **FYI.** Each recipient's seen-signal is `👀` — the same reaction primitive `[ACK]` already uses for its awareness attestation. A comment is welcome when a recipient has something to say, but is never required. The initiator checks the reaction list to see who has seen it, then closes as ordinary housekeeping — no resolution comment required, since the reaction list already carries the "who saw it" evidence and GitHub's timeline preserves it.
- **Discussions that need a conclusion** (Q&A, Ideas, or any thread opened to converge something). The initiator decides when it is concluded: they write a summarizing final comment, then close. GitHub's native Q&A obligations — the author accepts an answer, then `/relay:settle` closes it (`close-answered-question`) — are the reducer-visible special case of this same rule, not a separate one. For every other Discussion category this is a documented convention: no new reducer signal exists, and none is added by this ADR.

The unifying principle is simpler than the alternative the design started from: closure ownership does not depend on the object's category or its number of addressees. Every Discussion has exactly one initiator, and that initiator is always the closer. What differs between FYI and a converging Discussion is only whether a final comment is required before closing — not who is allowed to close.

## Considered and rejected

**(i) Recipient-closes-FYI.** An earlier draft of this ADR had a single-recipient FYI closed by its own recipient (close = "seen"), with a multi-recipient FYI deferring to the initiator so a first responder couldn't hide the notice from the rest. Rejected: it split closure into two rules gated on recipient count, which a reader — and an agent applying the rule — has to disambiguate per object before acting, and the split existed only to patch the multi-recipient race that the initiator-closes design avoids by construction. One rule that already handles both counts is simpler than two rules plus a boundary condition between them.

**(ii) Comment-as-required-seen-signal.** An alternative made a comment the FYI recipient's seen-signal instead of `👀`, on the theory that a comment is more informative. Rejected: Relay already has an awareness-reaction primitive (`👀` on `[ACK]` Announcements); requiring a comment for FYI would duplicate that primitive under a different rule for a nearly identical need, and a required comment raises the per-recipient cost of acknowledging an FYI — from one click to composed text — which lowers compliance rather than improving signal quality. `👀` stays optional-comment/required-reaction for FYI, matching ACK.

**(iii) Atemporal engagement suppression.** The first working draft of this ADR's reducer change suppressed a `mentioned-in-prose` notice on ANY prior viewer engagement with the object, with no notion of *when* — commenting on a thread once, ever, silenced every future mention in it. Rejected on judgment-seat review (a fable pass caught this before it shipped): this re-opens ADR-093's founding scenario. A prose "@person please respond" left in a thread the person had once, long ago, commented in would never surface as a notice again, no matter how much later or how unrelated to their old comment it was — exactly the silent-notification failure the notices tier exists to prevent. The fix is the **temporal** rule below: suppression for a comment-borne mention is scoped to what the viewer has actually had a chance to see, not to whether they have ever visited the thread at all.

## Reducer change

Unlike an earlier draft of this ADR (which stated no reducer change), this decision requires one: `compute-state.mjs`'s `mentioned-in-prose` notice suppression now also fires on viewer engagement, not only when the object closes — but the rule is **not one rule**, it is two, split by where the mention lives:

- **Title/body mention — atemporal.** No per-mention timestamp exists for a body/title edit, so the only available signal is "has the viewer ever engaged with this object at all" — any comment the viewer authored on it, or (Discussions only) their own `👀` reaction (`viewerEngaged()`, reusing the ACK path's existing `EYES`-reaction data via `hasEyesFrom`; no new query field for this half).
- **Comment-borne mention — temporal.** A mention inside a comment carries that comment's own `createdAt` (newly added to the GraphQL selection for Discussions and PRs — Issues already had it), so suppression compares timestamps: a comment-borne mention is suppressed only when it is at-or-before the viewer's own last comment on the object (`viewerLastCommentAt()`, the max `createdAt` across the viewer's own comments there, or `null` if they never commented). **A mention that arrives after the viewer's last comment still fires** — string-compared as ISO-8601, the same way the reducer already compares review/revision timestamps elsewhere. `👀` deliberately does **not** feed this comparison: it carries no timestamp in the collected data, so it can never cover a comment-borne mention, only a title/body one. A comment with no `createdAt` at all (a legacy fixture predating this field) is treated as older than everything, via defaulting to the empty string in the string comparison — so it is suppressed once the viewer has commented on the object at all, keeping old fixtures meaningful instead of silently never-suppressing.

**Deliberate asymmetry, unrelated to the temporal split above, stated once so it isn't mistaken for an oversight:** Issues and PRs carry no reaction query in the current GraphQL collection (`graphQlQuery()`), so only comment-engagement ever suppresses their mentions — a bare `👀` on an Issue or PR is not fetched and cannot suppress anything there, title/body or comment. Comment-engagement already covers the suppression need for those two types, so the collector was not extended to fetch reactions there too.

`compute-state.test.mjs` grew from 59 (pre-ADR-096) to 66. The ADR-096-v1 pass added four (FYI + viewer-comment → no notice; Discussion + viewer-👀 → no notice; Issue + viewer-comment → no notice, demonstrating the reaction-query asymmetry; uninvolved mentioned viewer → notice still fires) and updated one pre-existing test in place to the (then-atemporal) new outcome. This v2 pass added three more testing the temporal split specifically (a stranger's comment mention followed by the viewer's own *later* comment is suppressed; a viewer's `👀` never suppresses a comment-borne mention even when it precedes the mention; a comment mention with no `createdAt` is suppressed once the viewer has commented at all) and **restored** that one pre-existing test to its original pre-ADR-096 expectation — a stranger's comment-borne mention that arrives *after* the viewer's own earlier comment now fires again, with an updated comment explaining why the atemporal interim answer was wrong. All other suppression tests (F1, F2(b), F3, F4's self-mention case) needed no change and stay green throughout.

## What this does NOT change

- **No schema version bump.** `digest`'s `schemaVersion` stays at 3 — the notice-suppression logic and one GraphQL field changed, not the shape of the output.
- **No obligation change.** FYI remains obligation-free under every reading; a non-Q&A Discussion still creates no obligation, only an initiator-owned convention for when it is judged concluded.
- **No ACK lifecycle change — as of THIS ADR.** `[ACK]` Announcements kept their existing rule (designated recipient's `👀` completes the round; closure is optional housekeeping), distinct from the FYI rule above (an ACK is not an FYI; they use the same reaction primitive for different purposes). **This claim is superseded by ADR-097**, which makes every named recipient on a non-`fyi` Announcement receipt-owing by default — the single-designated-recipient reading this ADR left alone no longer holds.

## Consequences

- `plugins/relay/skills/report/SKILL.md` states the FYI closure rule (initiator closes; recipients signal `👀`) at its existing recipient-tagging bullet.
- `plugins/relay/skills/settle/SKILL.md` names the Discussion initiator as the closure owner for every Discussion, with the FYI branch requiring no resolution comment and the converging-Discussion branch requiring one.
- `plugins/relay/skills/reply/SKILL.md` distinguishes FYI `👀` (a seen-signal the initiator later reads) from ACK `👀` (an awareness attestation that completes an obligation) at its existing discipline bullet.
- `plugins/relay/skills/digest/SKILL.md` documents the atemporal/temporal split, the reaction-query asymmetry, widens "three suppression cases" to four, and states plainly that a mention arriving after the viewer's last engagement fires while one arriving before does not.
- `plugins/relay/CLAUDE.md`'s lifecycle matrix states the initiator as FYI's obligation-owner ("None") with closure ownership moved into the completion/closure column so that column stays single-meaning; the closure-semantics paragraph and the obligations-versus-notices paragraph both gain the engagement-suppression sentence.
- `docs/design/relay.md` gains a revised closure-semantics paragraph and a changelog line recording the reducer change and the (now three) rejected variants.
- `compute-state.mjs`'s `graphQlQuery()` gains `createdAt` on the Discussion and pull-request comment selections (Issues already had it); the reducer gains `mentionsWithSource()` (replacing `mentionsOfOthers()`) and `viewerLastCommentAt()`, and the notice-suppression check in `reduceObligations` now branches on mention source. `compute-state.test.mjs` goes from 59 to 66 tests: four added in the v1 pass, three more added and one restored to its original expectation in this v2 pass, all green.
- `relay` moves `1.3.0` → `1.4.0` — new documented process rules plus one reducer behavior change, no schema break, so a minor bump.
- No new GitHub object type, schema field beyond `createdAt` on two existing selections, or daily verb. The daily verb set stays `launch · report · digest · reply · brief · settle` plus explicit-only `migrate`.
