# ADR 067 — Dispatch tiers + consultant seat (capability slot)

**Status**: accepted
**Date**: 2026-07-13
**Extends**: [ADR-058](docs/adr/058-shape-cost-tiers.md) · [ADR-059](docs/adr/059-cost-tier-marketplace-wide.md) (a skill's own declared cost tier) with a second dimension — who gets *dispatched* to do a piece of work, independent of any single skill's tier.
**Source**: `blueprints/thoughts/2026-07-13-dispatch-tiers-consultant-seat.md` (four forks, decided by Paul the same session); folds in `blueprints/thoughts/2026-07-13-rule7-second-door-advisor-escalation.md`'s advisor-escalation proposal as this ADR's consultant-seat resolution.
**Amended**: [ADR-087](087-consultant-seat-drops-advisor-rung.md) (2026-07-17) removed the `advisor`-tool first rung from the resolution order — the seat's role-not-tool principle stands; the tool under it proved effectively never available. [ADR-089](089-consultant-seat-no-self-escalation.md) (2026-07-17) capped escalation at one hop — a sub-agent dispatched into the seat does not re-invoke the resolution order on its own stuck-ness.

## Context

Two previously-separate questions turned out to be one principle. First: rule ⑦ ("below 90% confidence → ask") only had one door — ask the user — but some stuck judgment calls (which refactor cut is deeper, is this really a God Function) have an answer that doesn't live in the user's head at all; asking them just passes the ball to someone who can't catch it. The proposed second door was Claude Code's `advisor` tool — call a stronger model with the full transcript. Second, separately: strong-model sessions (Fable) routinely delegate reconnaissance and mechanical execution to a cheap sub-agent (Sonnet), and the reverse also happens — a cheap session needs to escalate a stuck judgment call to a stronger reviewer.

Both turned out to be the same principle stated at different grains: **strong reasoning sits on the judgment seat (planning, review/check, unblocking); cheap capacity does reconnaissance and disciplined execution; downgrading is safe only when a strong-seat checkpoint reviews the output downstream.** The `advisor` tool is just one *implementation* of "escalate to a stronger seat" — and a fragile one: this session's own hands-on test found `advisor` unavailable all session despite correct settings and a bypassed experimental flag, with no mid-session fix possible. A mechanism that only works by naming a specific tool breaks when that tool is unavailable; a mechanism that names a *role* survives the tool changing under it.

## Decision — four forks, one principle

1. **The consultant seat is a capability slot, not a hardcoded tool call** (mirrors shape's `browser-verify` slot pattern). Resolution order: ① the `advisor` tool, if available (cheapest — it sees the full transcript for free) → ② a fresh strong-model sub-agent, briefed explicitly → ③ the seat is empty: fall back to rule ⑦'s original door (ask the user; mark uncertain, never assert).
2. **The sub-agent consultant's posture depends on the kind of stuck.** Not yet decided between two options → dispatch as a **neutral reviewer**: hand over the question plus both sides' evidence, withhold which way you're leaning, let it recompute independently. Already leaning but not confident → dispatch as a **devil's-advocate**: state the current lean, command it to try to break it. The hard rule either way: **never feed the reviewer your own conclusion when the point is an independent check** — that turns it into an echo chamber.
3. **Escalation is symmetric and injects durable artifacts, not paraphrases.** A strong session downgrades to a cheap hand (routine); a cheap session can equally escalate to a strong seat (the `advisor` tool is the natural first choice here, since it alone sees the whole transcript for free — a cheap-session-primary-plus-strong-advisor pairing is the tool's own intended use). Escalating has a failure mode downgrading doesn't: **the party picking what evidence to show the stronger reasoner is the weaker one**, and may pick the wrong thing. Mitigation: inject the **path to a durable artifact** (a plan file, a diff, the actual files) so the stronger reasoner re-reads the primary source itself, never a summary produced by the picker.
4. **No new scout skill.** Dispatching reconnaissance (read code, search the web) is a plain use of the `Agent` tool, not a protocol that can be gotten wrong in a way worth codifying — the skills that already need disciplined recon (`/nav:audit`'s domain fan-out, `deep-research`'s verification loop) already own that discipline. The only thing worth standardizing is a one-line reporting convention: **recon sub-agents report grounded in fact (file:line / source URL), not impressions.**
5. **Naming is relative, not tool-pinned**, so swapping either seat changes nothing but one line. The judgment seat is always "the session model" in prose (swap Fable → a future model = change harness, zero file edits — the plan-artifact convention already makes `blueprints/plans/*.md` the cross-vendor seam). The cheap hand is always "the tier the root `CLAUDE.md` table names" (currently `model: sonnet`) — swapping it is a one-line edit to that single table, not a hunt through every skill file. A dispatcher may override the default tier for a single step on the spot when that step turns out to be unusually judgment-dense — the default is a convention, not a lock.

## Why

- Naming the consultant seat by role rather than by tool is the direct lesson from this session's own `advisor` outage: the mechanism has to degrade gracefully to "ask the user" rather than silently doing nothing when the specific tool is unavailable.
- The reviewer/devil's-advocate split matters because a sub-agent only knows what it's told — feeding it a pre-formed conclusion collapses the check into confirmation. This is the same discipline `research:critique`'s self-attack loop already depends on, generalized to any consultant dispatch.
- Injecting a durable artifact's path rather than a summary closes the "the picker might pick wrong" gap that a naive escalation protocol would otherwise leave open — the stronger reasoner re-derives from the primary source instead of trusting the weaker session's framing of it.
- Not building a scout skill follows the same razor that retired `nav:doctor` (ADR-021): a wrapper around something that isn't a protocol-with-discipline is ceremony, not convenience.

## Consequences

- Root `CLAUDE.md`'s Authoring conventions gains a "★ Dispatch tiers" bullet stating the principle, the tier default line, the consultant-seat resolution order, and the recon reporting convention — the single owner other files reference rather than restate.
- Three dispatch points each gain a one-line instance pointer to the root convention (no restatement): `plugins/nav/skills/plan/SKILL.md` (Stage 4's sub-agent option), `plugins/shape/skills/build/SKILL.md` (the Scheduling section's dispatched item agents), `plugins/nav/skills/audit/SKILL.md` (the Mode-3 domain fan-out, D2).
- `blueprints/thoughts/2026-07-13-rule7-second-door-advisor-escalation.md` gains an appended "軟依賴條款" (soft-dependency clause) section — advisor is the resolution order's *first* rung, not a required step; assume it may be unavailable mid-session with no fix; fall back to rule ⑦'s original door. The rest of that thought is left untouched; this ADR folds its proposal in rather than superseding it outright.
- `blueprints/thoughts/2026-07-13-dispatch-tiers-consultant-seat.md`'s status line is updated to record graduation to this ADR.
- **Not decided here**: a quantified threshold for "this single step is judgment-dense enough to escalate on the spot" (left to the dispatcher's judgment, per the source thought's own open question) — and whether a validator should gate the 6 `model: sonnet` frontmatter values against the tier table (the "frontmatter value ≠ table value" drift risk the source thought flags but leaves for implementation time).
