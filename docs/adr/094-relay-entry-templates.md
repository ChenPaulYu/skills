# ADR 094 — GitHub entry-point templates extend recipient-tagging to organic traffic

**Status**: accepted
**Date**: 2026-07-21
**Refines**: [ADR-090](docs/adr/090-relay-github-native.md), [ADR-093](docs/adr/093-relay-obligations-vs-notices.md)

## Context

Relay 1.1.0 (ADR-093) closed the gap where a prose `@mention` typed straight into an existing GitHub thread reached no one's digest, by adding a non-binding `notices` tier and by ratifying that every `report`-created object — FYI included — names an explicit `@recipient`. Both fixes are real, but both operate at the same layer: the **router**, `/relay:report`. A human who opens a GitHub Issue or Discussion directly, without going through `report`, never passes through that router at all. Nothing on the GitHub side nudges them to name a recipient, pick the right shape, or state a completion condition — the object's `title`/`body` boxes are free text, and a viewer typing into them has no reason to know Relay's routing rules exist.

A first-principles pass on this gap asked what capture actually is, rather than assuming it must live in Relay's own code. Capture is a property of **signals plus entry**, not of storage: `digest`'s reducer already computes obligations and notices correctly once GitHub state exists (assignment, requested review, `[ACK]` + `👀`, `isAnswerable`/`isAnswered`) — the shortfall is entirely upstream, at the moment an organic Issue or Discussion is *created*, before any of that state exists to reduce. GitHub itself already owns a write-time enforcement layer for exactly this moment: Issue Forms (`.github/ISSUE_TEMPLATE/*.yml`, with `validations: required` and `blank_issues_enabled: false`) and Discussion category forms (`.github/DISCUSSION_TEMPLATE/*.yml`). Both are native GitHub features that require specific fields before an object can be submitted at all — no Relay code runs, and no Relay database records anything; GitHub's own form validation is the enforcement.

This enforcement binds GitHub's **web UI** creation path only. `gh issue create`, `gh api`, and direct GraphQL mutations bypass form validation entirely — and agent-created traffic is exactly API traffic, not web-UI traffic. That is precisely why router discipline in `report/SKILL.md` still owns every object Relay itself creates; the template layer below reaches a different, non-overlapping population (a human typing directly into the GitHub web UI), and is a convention nudging that population, never a contract binding agents or API callers.

## Decision

`/relay:launch` installs and audits a template layer that extends the explicit-recipient rule to organic (non-`report`) traffic:

- `.github/ISSUE_TEMPLATE/task.yml` and `decision.yml` — Issue Forms requiring an "Owner" / "Decision owner" `@handle` field, so an organically-opened Issue at least names who it's for even before assignment happens.
- `.github/ISSUE_TEMPLATE/config.yml` — `blank_issues_enabled: false`, closing the escape hatch that lets a blank, template-free Issue bypass the forms above (write-access accounts retain a "Maintainers only" blank option per GitHub's own model; that is a GitHub platform limit, not a Relay gap).
- `.github/DISCUSSION_TEMPLATE/announcements.yml` and `q-a.yml` — Discussion category forms requiring a "Recipients" field (Announcements) naming at least one `@handle`, and offering an optional "who's best placed to answer" field (Q&A) that is explicitly documented as producing a notice, not an obligation.
- `.github/pull_request_template.md` — a short checklist nudging the author to request a reviewer at creation, matching the `request-reviewer` obligation ADR-093 already made a hard invariant.

A Discussion category form's filename must equal that category's **live slug** or GitHub silently never serves it — a mismatch renders no error, it simply never appears. GitHub derives a slug from a category's name at creation time, and slugs are per-repository: a repository may rename, add, or remove categories independently of any other repository's defaults. `q-a` (not `q-and-a`) is GitHub's own default slug for the Q&A category — the first cut of this ADR shipped `q-and-a.yml`, which is a real filename but not a real slug, so it was silently inert in every repository using GitHub's default category names; a live probe against `launch`'s own audit output caught the mismatch. The reference filenames therefore assume GitHub's own default category names ("Announcements", "Q&A"); a repository that renamed a category, or never enabled Q&A, needs a different filename or none at all, and `launch`'s audit reads the repository's actual live slugs (GraphQL `repository.discussionCategories.nodes{name,slug}`) rather than assuming the defaults hold.

Template assets live at `plugins/relay/skills/launch/references/templates/`, the same directory shape GitHub expects under `.github/`, so `launch` can copy them into a target repository verbatim. `launch`'s existing audit → gated mutation → read-back-verify flow applies unchanged: template presence and content are read back like any other setting, missing or drifted templates enter the mutation plan, and nothing is written without the standard approval gate. A `DISCUSSION_TEMPLATE` filename matching no live slug is a distinct audit finding, `inert (slug mismatch)`, never folded into `missing` — the two have different fixes (write a new file vs. rename/remove an existing one) and conflating them would hide a template that exists on disk but is dead to GitHub.

**Templates are a convention, not a contract**, in the ADR-071 sense: the digest reducer (`compute-state.mjs`) never reads a template's field values, and no new machine gate lints them. A repository without these templates keeps working with Relay exactly as it did in 1.1.0 — organic traffic there still lacks the nudge, but nothing breaks, and `launch` self-reports the absence as a `missing` finding rather than a blocker. Templates only ever influence what a *human* is prompted to type before an object exists; they cannot enforce anything once the object is created, and Relay's obligation semantics remain entirely GitHub-native (assignment, `👀`, requested review, `isAnswerable`/`isAnswered`) exactly as ADR-090 established.

## Rejected alternatives

**A thought-file hybrid — dual-writing template-submitted fields into a parallel Relay-owned store.** Once a template captures an explicit "Owner" or "Recipients" field, it's tempting to have `launch` or a webhook mirror that value somewhere Relay controls, so recipient-tagging could be verified independently of GitHub's own object state. This is the same parallel-database shape ADR-090 already rejected wholesale — "no parallel database, Relay status frontmatter, thought stream, decision ledger, or separate content repository in v1" — for the same rule-① reason: a fact written in two places drifts, and the dogfooding history that motivated ADR-090 in the first place was full of exactly that drift. GitHub's own Issue/Discussion body already *is* the durable record of what a template field contained; a second copy would only add a place for it to go stale.

**Reducer-side parsing of template field values.** Instead of treating templates as a write-time prompt, `compute-state.mjs` could parse a submitted Issue/Discussion body for the template's field markers (GitHub renders form fields as `### Owner\n\n@handle` in the body) and derive obligations from them directly. This was rejected because it promotes a convention to a contract without a machine gate protecting it: a hand-edited body, a repository using its own template variant, or GitHub changing its rendering format would silently break parsing with no validator to catch it — exactly the distinction ADR-071 draws between the three real contracts (relay thought frontmatter, the manifest/generated-artifact set, `docs/core/`'s freeze protocol) and everything else, which stays a tolerant, best-effort convention. The reducer already has a correct, fully GitHub-native source of truth for every obligation it computes (assignment, review request, `👀`, `isAnswerable`); template body text would be a second, weaker path to the same facts, not a new capability.

## Consequences

- New reference assets, not consumed by any script: `plugins/relay/skills/launch/references/templates/{ISSUE_TEMPLATE/{task,decision,config}.yml, DISCUSSION_TEMPLATE/{announcements,q-a}.yml, pull_request_template.md}`.
- `launch/SKILL.md`'s audit checklist gains a live-slug-aware template-presence check (`ready` / `missing` / `inert (slug mismatch)` / `blocked` / `cannot verify`); its mutation-gate step gains "copy from `references/templates/`, show contents, read back" as the standard template mutation; a new "Entry-point templates" section states the convention-not-contract boundary and the per-repository-slug caveat explicitly.
- `report/SKILL.md` gains one cross-reference sentence: organic traffic is nudged toward the same recipient-tagging rule by the repository templates `/relay:launch` installs.
- `plugins/relay/CLAUDE.md` gains a short mention of the template layer in the object-router section, plus this ADR in its refines list.
- `docs/design/relay.md` gains a short paragraph plus a dated changelog line.
- `compute-state.mjs` and its 59-test suite are untouched — this ADR adds no new obligation type, blocker, or schema field, and `SCHEMA_VERSION` does not move.
- No new GitHub object type, parallel state store, or daily verb. The daily verb set stays `launch · report · digest · reply · brief · settle` plus explicit-only `migrate`; `launch` absorbs this as an extension of its existing audit/configure surface.
- relay `1.1.0` → `1.2.0` (a new audited/configured surface is a minor addition, not a patch).
