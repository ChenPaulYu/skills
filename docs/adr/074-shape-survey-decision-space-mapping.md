# ADR 074 — `shape:survey`: decision-space mapping, diff not teaching

**Status**: accepted
**Date**: 2026-07-14
**Source**: `blueprints/thoughts/2026-07-14-elicit-ecosystem-four-quadrants.md` (the four-quadrant design rulings) · `blueprints/plans/2026-07-14-elicit-ecosystem.md` (Phase 1 of the implementation plan)
**Phase**: 1 of 4 (`survey` → `probe` → elicit gatekeeper upgrade → behavioral testing); this ADR covers `survey` only.

## Context

`/shape:elicit` converges a conceptual decision by drawing it out of the user through a grounded grill — but that engine has one load-bearing assumption: the answer already lives in the user's head, waiting for friction to surface it. The four-quadrant analysis (`blueprints/thoughts/2026-07-14-elicit-ecosystem-four-quadrants.md`) named the knowledge-state map elicit actually serves:

| Knowledge state | Solved by | Verb |
|---|---|---|
| I know I know | Draw it out and align | `shape:elicit` (existing) |
| I know I don't know (the world knows) | Map → fill the gap → re-align | `shape:survey` (this ADR) |
| I don't know I don't know (the world knows) | elicit detects it mid-grill → offers survey | elicit's gatekeeper role (state 2 or 3) |
| Nobody knows | Design a minimal experiment | `shape:probe` (Phase 2, not covered here) |

elicit only ever served state 1. States 2 and 3 — a known-unknown the user can name, and an unknown-unknown that only surfaces as a stuck grill — had no verb: grilling a blank spot produces nothing to push against, so calling elicit into that gap just burns turns without converging anything.

## Decision

1. **`survey` is a new `shape` skill whose identity is mapping, not teaching.** The deliverable is the **diff** between the user's stated understanding of a decision space and the full map of that space — axes and options, grounded against the real repo. Teaching (explaining a found gap) is a service performed *on* that diff; it is never the product on its own. A survey that hands back a generic checklist, ungrounded from the user's actual situation, is a textbook any chatbot can produce — not worth a dedicated verb. This is the identity ruling from the thought file, restated here as the skill's non-negotiable core.
2. **Two entrances, because "summon a skill to reveal your own blind spot" is a paradox.** Nobody spontaneously calls for a skill whose entire point is surfacing something they don't know they're missing. The fix isn't making survey auto-fire (that violates the family's summoned-not-automatic law); it's giving it **two doors**: (a) the user summons it directly when they can *name* the terrain they're shaky on (state 2 — a known unknown), and (b) `/shape:elicit` *offers* it mid-grill when a volley is stuck because the user is missing a whole axis they never named (state 3 — an unknown unknown detected by the one skill already in the room). The detection logic lives in elicit (Phase 3, a later ADR); the mapping engine lives here, once, never duplicated.
3. **The map is built in two layers, borrowing `frame:orthogonal`'s independence discipline by protocol, not by call.** Domain-prior candidate axes only earn a place in the map if they pass orthogonal's test — move one, the others must not move; a coupled "axis" is really a checklist item wearing an axis's clothes, and gets merged or re-cut. A reality-diff layer then grounds each surviving axis against the actual repo/existing decisions at `file:line` resolution — an ungrounded axis is still a guess. This mirrors the meta-skill borrow-by-protocol pattern established for `shape:position` (ADR-008: borrow elicit/mockup/first-principles by protocol, never by invoking another skill) — skills in this marketplace never call each other; they restate the discipline they need.
4. **The diff separates "missing whole axis" from "missing point on an axis."** These are different kinds of gap that route differently — a missing axis usually needs more explaining than a missing point on an axis the user already knows exists — so the report keeps them apart rather than merging into one undifferentiated gap list.
5. **Gap-filling is sized to the gap and never calls a sibling directly.** A small gap is explained in place (plain language + analogy, no hand-off). A big gap gets a **guarded, one-shot offer** — never a call — to `/frame:analogize` (when the gap is a concept needing deliberate transfer) or `deep-research` (when the gap is external-world knowledge needing co-read sources). This keeps survey inside the family's offers-not-calls law (ADR-007/015) and keeps `analogize`/`deep-research` each doing the one job they already own, rather than survey growing a duplicate teaching engine (the same N+1 razor that kept diagnostic-mode root-causing inside `elicit` instead of a parallel `diagnose` skill).
6. **Reconnaissance is dispatchable; map-quality judgment is not.** Per the dispatch-tiers convention (ADR-067), the repo-scanning/doc-lookup legs of building the reality-diff layer can go to a cheap-tier sub-agent, reporting grounded in fact (`file:line` / URL). Which candidate axes survive the independence check and what belongs in the diff is the map-quality judgment call, and stays with the session model.
7. **Output is a `blueprints/thoughts/<date>-<topic>-survey.md` doc, tolerant-reader compliant (ADR-071).** `blueprints/` is a convention, not a contract: survey scaffolds the tree if absent (mirroring `align`'s first-run behavior), tolerates a non-standard shape, and self-reports which tier it read/wrote from. The doc states explicitly that it is input to `/shape:elicit` — the survey's diff is what elicit grills next, not a standalone reference. Write-gated: the doc's content is shown before it's written; everything upstream of that (the map, the diff, the fills) is read-only.

## Why

- **Mapping-not-teaching as the identity, not an incidental scoping choice.** If survey's deliverable were "explain the space," it would collapse into a generic tutorial generator with no way to bound scope or ground itself in the user's actual situation — exactly the failure the four-quadrant thought named and rejected. The diff *is* the value: it's grounded in what the user specifically doesn't know, not what a topic generically contains.
- **The next-doctor razor (ADR-021) applies to "teach" directly, which is why teach was *not* built as its own verb.** "Grill me on this" already differs from "explain this to me" — the thought's third ruling found that a standalone `teach` skill would just be "call `frame:analogize` for you," too thin a wrapper around a capability that already exists as its own verb. Folding gap-filling into survey's step 3 (as an offer, not survey's own re-implementation) avoids building that thin orchestrator.
- **Two entrances resolve a real invocation-paradox, not a hypothetical one.** A skill whose entire point is surfacing an unknown-unknown cannot rely solely on user-summon (state 3 by definition means the user doesn't know to summon it) — parking detection inside elicit (the one skill already present when a grill gets stuck) is the only place the paradox has a resolution that doesn't require a second standing engine watching every conversation.
- **Borrowing `orthogonal` by protocol, not by call, keeps the family's hard invocation law intact.** Skills in this marketplace never invoke each other (stated repeatedly across `shape` and `frame` CLAUDE.md files); the established workaround — restate the discipline, credit the source — is what `shape:position` already does with elicit/mockup/first-principles (ADR-035/041), and what `shape:probe` (Phase 2) will do with dialectic/first-principles/orthogonal in sequence.
- **The dispatch-tiers split (recon cheap, judgment strong) matches every other instance of that convention in this repo** (`nav:plan` Stage 4, `shape:build` scheduling, `nav:audit` Mode-3 fan-out) — survey doesn't need a new rule here, just an instance pointer.

## Overlap / boundary analysis — four neighbors

- **vs `/nav:map`** — `nav:map` renders the *repo's structure* (modules, imports, the codebase graph) for navigating real code. `survey` renders the *decision space* (options/axes bearing on a choice), which may not correspond one-to-one with existing files. Structure map vs decision map — different objects entirely, so the trigger risk is low; the boundary sentence exists mainly to stop a reader from reaching for `nav:map` when what they actually want is coverage of their own understanding.
- **vs `deep-research`** — `deep-research` produces an external, cited research report on a topic, standalone. `survey` may *offer* `deep-research` for a big external-knowledge gap, but survey's own output is never a research report — it's a coverage diff against a decision space, and the offer is guarded/one-shot, never a call. The two only meet at that one hand-off point.
- **vs `/shape:elicit`** — the sharpest boundary, because the two are meant to chain. elicit draws an answer **out of** the user (the answer already exists in their head); survey maps **what the user doesn't have** (there's a real gap, not just unspoken clarity) and fills it with a grounded diff, never a grill. elicit is the consumer of survey's output and the sole door that offers survey mid-grill — the two-entrance design (decision 2) is what keeps this boundary from becoming a race condition where both skills think they own "the user seems stuck."
- **vs `/nav:audit` Mode 2** — Mode 2 is a read-only feasibility quick-check: can the *codebase* support a target spec? `survey` checks whether the *user's own understanding* of a decision space is complete before they decide anything — a knowledge-completeness question about a person, not a feasibility question about code. Different subject entirely (the user's map vs the codebase's capability), so the trigger overlap is near-zero; stated for completeness since both are "check something before committing" verbs.

Trigger-collision risk overall is low because each neighbor's core question differs from survey's ("is my repo navigable" / "what does the external literature say" / "draw out what I already believe" / "can the codebase support this spec" — none of these is "is my understanding of this decision space complete"). The description's boundary sentences (see `plugins/shape/skills/survey/SKILL.md` frontmatter) restate these four lines verbatim so a routing model sees them without opening the body.
