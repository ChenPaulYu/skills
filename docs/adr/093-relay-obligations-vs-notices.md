# ADR 093 — Relay separates obligations from notices, and gives Q&A native obligations

**Status**: accepted
**Date**: 2026-07-21
**Refines**: [ADR-090](docs/adr/090-relay-github-native.md), [ADR-091](docs/adr/091-relay-awareness-review-task-evidence.md), and [ADR-092](docs/adr/092-relay-native-lifecycle-completion.md)

## Context

An adversarial parity audit compared Relay's GitHub-native design against the legacy file-based system it replaced, specifically hunting for gaps a real user would feel. It found two.

First: a prose `@person please respond` written directly on a GitHub object reaches no one's digest. `report`'s router only governs traffic that is created through `report` — an `[ACK]` Announcement, an assigned Issue, a requested-reviewer PR. Nothing stops a counterpart from typing an informal ping straight into an existing Discussion, Issue, or PR comment instead of going through the router, and once they do, `digest`'s obligation contract has no signal for it: no ACK label, no assignment, no review request. The live casualty was a Discussion that closed unanswered — the mention inside it never surfaced anywhere actionable.

Second: Q&A produces zero digest output. `compute-state.mjs` fetched a Discussion's `isAnswered` field from the start but never read it in the reducer — there was no code path that turned an open, unanswered, or newly-answered Q&A Discussion into anything a viewer would see. A Discussion author with an accepted-but-still-open question, or an unaccepted answer sitting for weeks, got nothing from `digest`.

## Decision

`digest` now returns two separate tiers instead of one.

**Obligations** keep the existing real-obligations-only contract: work owed, with a named owner and a native GitHub completion signal (ACK reaction, assignment, review request/verdict, accepted answer, approved+enforced merge, or a reviewer request). Nothing changes about what counts as an obligation except the additions below.

**Notices** is a new, separate array. A notice is awareness — worth looking at, never work owed — and must never be presented, sorted, or counted alongside an obligation. One kind:

- `mentioned-in-prose` — the viewer's login is `@mentioned` anywhere in the title, body, or a comment of an open Discussion, Issue, or PR, with no obligation signal already on that object. Suppressed in three cases (a fable review of the first cut found the first two missing — see "Suppression is not just obligations" below): the object already carries an obligation for the viewer; the object already carries a blocker for the viewer; or the object carries a **formal signal** for the viewer (designated ACK recipient, requested reviewer active-or-history, or assignee) regardless of whether that round is already complete. A mention is also never counted from text the viewer authored themselves.

**Q&A gets native obligations**, driven directly by GitHub's own `isAnswerable`/`isAnswered` fields rather than an invented state: an open, answerable Discussion the viewer authored, unanswered, with someone else's comment on it, is `DECIDE/ACT accept-answer-or-follow-up`; the same Discussion once accepted and still open is `SETTLE close-answered-question`. A viewer who is only mentioned on someone else's question — not the author — gets the `mentioned-in-prose` notice instead; that traffic was never theirs to obligate on.

**Router discipline is stated explicitly** in `report`: an ask that expects a response must land in one of the three natively-owned shapes (assigned Issue, requested-reviewer PR, or a named-recipient `[ACK]` Announcement). Anything else is FYI by definition. This does not replace the notices tier — it reduces how often prose pings happen in the first place, while notices catch the ones that happen anyway on organic GitHub traffic the router never touched.

Two rules were ratified as hard invariants during this design, not left as soft guidance:

1. **Every report explicitly tags a recipient — all routes, FYI included.** A pure FYI Announcement now must name at least one explicit `@recipient` in its title or body; an untagged FYI is a broadcast into the void. The notices tier is exactly what makes this safe rather than redundant: it is the mechanism that guarantees a tagged FYI still reaches that person's digest (as `mentioned-in-prose`), even though FYI itself carries no obligation. Tagging without the notices tier would be theater; the notices tier without tagging would have nothing reliable to catch.
2. **Anything needing review is captured as an obligation, never a notice.** `no-reviewer-requested` — first designed as a notice (a self-nudge) — is reclassified as a real `DECIDE/ACT request-reviewer` obligation for the PR author. This makes an explicit invariant true: **an open, non-draft PR is never obligation-free.** At every moment, either a reviewer owes a verdict, the author owes changes after `Request changes`, the settlement owner owes a merge, the author owes a reviewer request, or a `settlement-owner-cannot-merge` blocker is visible. A draft PR stays exempt — it is not yet asking to be reviewed. The condition is precise, not a raw review count: it fires only when there is no active review request, no current-revision verdict that either satisfies required approval or comes from a historically-designated reviewer, *and* no historically-designated reviewer (active or withdrawn request) has a verdict on any other revision — that reviewer's stale verdict routes a re-review obligation to them instead, and must not also silently satisfy this one. A `COMMENTED` review never counts as a claim, from anyone; an early implementation used a raw `reviews.length === 0` check, which a drive-by comment or an undesignated party's stale approval could falsify — a live probe caught both, and the fixture-level tests originally used the same overly-permissive shape the code did, so nothing caught it sooner. A follow-up review then found the "any current-revision verdict counts" reading was itself still too broad: on a protected repo whose aggregate is `REVIEW_REQUIRED`, a never-requested volunteer's current-revision `APPROVED` is not sufficient by GitHub's own rules, so it must not read as a claim either — the check now only accepts a current-revision verdict when it actually satisfies `requiredApprovalSatisfied` or comes from a designated reviewer, closing that hole with no new prose exception, just a corrected condition. Demoting "nobody's been asked to review this" to mere awareness would have left exactly the kind of silently-stalled PR the notices tier exists to prevent elsewhere; the fix is to keep it out of notices entirely, not to widen what notices are allowed to mean.

   A follow-up fable review probed the invariant's own prose for over-claiming and found three real holes, each closed honestly rather than hidden: an `fyi`-labeled non-draft PR *is* deliberately exempt — the whole obligation block is skipped for it, same as any FYI object, so this is a named carve-out, not a bug; a viewer who owns settlement with satisfied approval but personally lacks merge authority (`viewerCanSettle` false) previously produced zero obligations *and* zero blockers, because per-viewer data can never reveal whether some other account could still merge it — the fix surfaces this as a visible `settlement-owner-cannot-merge` blocker instead of silence, closing the author's own side honestly while stating plainly that another account's authority is undetectable from here; and a ghost/deleted PR author has no native owner to route any obligation to, so such a PR sits outside the invariant entirely — a rare, named edge, not a silent gap.

### Suppression is not just obligations (fable review, second pass)

A live probe on the shipped 1.1.0 code found the `mentioned-in-prose` suppression rule too narrow: it checked only whether the *viewer's obligations* covered the object, not whether a *formal signal* for the viewer existed regardless of completion state. Concretely, a designated ACK recipient who had already added `👀`, or a reviewer who had already submitted their current-revision verdict, kept getting a `mentioned-in-prose` notice on that same object forever — because completing the round removes the obligation, but the object still names them in its text. Since every `report`-created object now tags an explicit recipient (rule 1 above), *every completed round* was on a path to becoming standing notice noise for exactly the person who finished it — the identical failure mode ADR-090 retired the old startup-digest hook to escape, reintroduced by this ADR's own fix.

The ratified suppression rule is now three-part: a mention notice is suppressed when the object carries an obligation for the viewer, when it carries a blocker for the viewer (a mentioned author of a blocked Core PR should see the blocker, not the blocker *and* a redundant notice), or when it carries a formal signal for the viewer — designated ACK recipient, requested reviewer (active or withdrawn history), or assignee — independent of whether that round is complete. This matches `digest/SKILL.md`'s own pre-existing definition of what a notice excludes ("no ACK, assignment, or review request"); the code had simply not implemented that definition fully.

A fourth, smaller gap in the same probe: text the viewer authored themselves — their own comment, for instance — that happens to `@mention` their own login produced a self-notice. Naming yourself in your own text is not someone pinging you. The mention scan now excludes any source (title/body attributed to the object's author; a comment attributed to that comment's author) that the viewer themselves authored, while still counting mentions from sources with no resolvable author. This carve-out was initially unimplementable for Issues specifically: the GraphQL selection fetched no `author{login}` for Issue nodes, so an Issue's title/body had no author to attribute against and a self-mention in one's own Issue body still self-noticed regardless of the fix above. The query now fetches it (it flows through the existing object spread with no further transform needed), closing that gap rather than leaving it as a known exception.

## Why not widen obligations instead

ADR-090's dogfooding already found false actionable items were the cost of treating every signal as an obligation — that is why `digest` stayed real-obligations-only rather than reflecting every notification. Folding prose mentions into `obligations` would reopen that exact failure mode: a mention is not a commitment from anyone, has no completion signal, and would sit in the list forever. Keeping it in a separate, explicitly-non-binding tier gets the visibility without inheriting that cost.

## Why not router discipline alone

Router discipline governs only what gets created *through* `report`. It cannot reach a `@mention` someone types directly into an existing thread's comment box — that traffic never passes through Relay's router at all. Stating the discipline in `report`'s SKILL.md reduces how often that happens by naming the right shape up front, but it cannot retroactively catch prose already on GitHub. Both are needed: router discipline at creation time, the notices tier as the safety net for everything that bypasses it.

## Recorded drops

Writing this ADR surfaced several previously-undocumented deliberate losses from the legacy file-based system. None are reversed by this decision; they are recorded here because they had no prior written home.

- **Legacy asset conventions** — the old system's file-naming, versioning, and compression conventions for attached assets have no GitHub-native replacement. Native attachments (drag-and-drop uploads, linked files) suffice for v1; the discipline around naming/versioning/compression is dropped, not ported.
- **Decision supersession linking** — when a closure supersedes an earlier object, naming the successor in the closing comment is now settle-practice, stated as one sentence in `settle/SKILL.md`. There is no separate linking mechanism or ledger; git history and GitHub's own reference syntax (`#123`) carry the trail.
- **`needs-input` label** — an assigned Issue labeled `needs-input` is a routing convention meaning the assignee is awaiting someone else's input before they can act. It does not create a second obligation; the assignment already is the obligation, and the label is context on it. Documented now in `digest/SKILL.md`.
- **Aggregate currently-effective-decisions view** — the legacy system's single view of "every decision currently in force" is retired in favor of on-demand cited briefs (`/relay:brief`) and direct citation of Decision Issues from Core PRs. No replacement aggregate view is planned; Raw GitHub plus briefs plus `core/` is the intended substitute.

## Consequences

- `compute-state.mjs`'s `reduceObligations` returns a new top-level `notices` array alongside `obligations`/`blockers`; `SCHEMA_VERSION` moves 2 → 3 as a semantic-tier contract change.
- The GraphQL collection query grows to fetch Discussion `category{isAnswerable}` (`isAnswerable` is a field of `DiscussionCategory`, not `Discussion` — a flat `isAnswerable` on the Discussion node is rejected by GitHub's schema), Discussion `author`, and comment bodies+authors (capped at 50) on Discussions, Issues, and PRs, plus PR `isDraft`. `normalizeLive` derives the discussion's usable `isAnswerable` from `category?.isAnswerable === true`, so a Discussion with no category (or `isAnswerable: false`) degrades to "not answerable" rather than crashing. The comment cap is self-reported via a top-level `commentScanTruncated` array rather than failing the run — unlike the harder page caps on discussions/issues/PRs, reactions, files, and review-request history, which still fail the collection hard.
- `report/SKILL.md` states entry-owner discipline explicitly, and its FYI refinement now requires an explicit `@recipient`; `reply/SKILL.md` and `settle/SKILL.md` gain the small cross-references needed to close the Q&A loop (accept/follow-up in reply, closure in settle) plus the one-sentence supersession-naming practice.
- `no-reviewer-requested` is an obligation (`DECIDE/ACT request-reviewer`), not a notice — the notices tier ends up with exactly one kind, `mentioned-in-prose`. Any future addition to notices must clear the same bar this one didn't: genuinely no owed action, only awareness.
- The blocked-collection output is exactly the declared schema (`schemaVersion, source, repository, viewer, blocked, blockers, obligations, notices`) — no collector-internal `objects` field leaks through, even though the input the collector hands the reducer carries one.
- A new blocker code, `settlement-owner-cannot-merge`, joins `core-enforcement-unverified` as a visible, named blocker rather than a silent zero-obligation state.
- Mention-notice suppression now checks obligations, blockers, and formal signals (ACK recipient / requested reviewer / assignee, regardless of completion) on the object, plus excludes self-authored mentions. `digest/SKILL.md`'s Notices section documents all three suppression cases and the self-mention carve-out.
- The Q&A `accept-answer-or-follow-up` stranger check ignores comments with no resolvable author (a deleted/ghost account is not evidence of a stranger). `commentScanTruncated` is scoped to open objects only — a closed object's undetected comments no longer matter to anyone's digest.
- No new GitHub object type, parallel state store, or seventh daily verb is introduced. The daily verb set stays `launch · report · digest · reply · brief · settle` plus explicit-only `migrate`.
