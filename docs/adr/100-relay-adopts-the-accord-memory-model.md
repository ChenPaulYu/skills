# ADR 100 — Relay 2.0.0 adopts the Accord memory model

**Status**: accepted
**Date**: 2026-07-22
**Refines**: [ADR-090](docs/adr/090-relay-github-native.md), [ADR-098](docs/adr/098-relay-upstream-nursery-routing.md), [ADR-099](docs/adr/099-relay-decision-reversal-consent.md)
**Supersedes**: [ADR-097](docs/adr/097-relay-receipt-default.md) in full; the Announcement-specific branch of [ADR-096](docs/adr/096-relay-closure-semantics.md); the awareness/ACK leg of [ADR-091](docs/adr/091-relay-awareness-review-task-evidence.md) and [ADR-092](docs/adr/092-relay-native-lifecycle-completion.md) outside legacy compatibility; the Announcement-specific hard rule and notices-scope framing of [ADR-093](docs/adr/093-relay-obligations-vs-notices.md)

## Context

Relay's obligation model grew, ADR by ADR, around one recurring object shape: an Announcement Discussion carrying an awareness receipt. ADR-091 split awareness from review from evidenced work; ADR-092 gave every event a native completion signal; ADR-093 added the notices tier and the open-PR invariant; ADR-096 ratified who closes a Discussion; ADR-097 flipped awareness from opt-in to receipt-default; ADR-098 named Discussions as Relay's upstream nursery; ADR-099 closed a consent gap in decision reversal. Each ADR was locally sound, but the accreted shape never had a single owning design: what a Decision *is*, where it lives once made, and how understanding gets derived from a pile of closed objects were never stated as one model — only inferred, ADR by ADR, from how obligations happened to compute.

A full-round convergence session between the repository owner and the session agent produced that missing design: [`blueprints/plans/2026-07-22-accord-memory-model.md`](blueprints/plans/2026-07-22-accord-memory-model.md) (the "blueprint"), ratifying four GitHub objects with clean boundaries (Discussion / Issue / PR / Commit), two memories (collaboration memory on GitHub, formal memory as versioned `decisions/`/`briefs/`/`core/` files), a Decision→Brief→Core disclosure stack with mechanical supersession, and a native three-tier metadata law (responsibility in native fields, semantics in structured prose, health in a conformance sweep). The blueprint is the design record for this wave; this ADR is the umbrella that adopts it, states what it supersedes, and gives the resulting break its version number.

The load-bearing simplification the blueprint makes: **there is no Announcement object.** A tell that needs a receipt is an Issue (an assignee confirms, then closes) — not a Discussion with a receipt bolted on. A fact that needs no receipt gets no object at all; a passing heads-up is a plain `@mention`, caught only by the notices tier. This directly removes the entire receipt-default apparatus ADR-097 built four months into Relay's life, and narrows what "awareness" means in the model to a legacy compatibility path.

## Decision

**Relay adopts the Accord memory model as its design basis, effective 2.0.0** — a semantic-break major version, since the Announcement/receipt machinery this releases removes is a real behavior change for any repository that relied on it. The blueprint governs; this ADR records what changes as a result and what does not.

### What retires

- **ADR-097 (receipt-default) retires in full.** No Discussion, `[ACK]`-titled or otherwise, produces a multi-recipient receipt obligation by construction any more. `announcementRecipients`, `isAnnouncementDiscussion`, the per-recipient `ACK` obligation loop, and the `close-announcement` SETTLE nudge (`receipts-collected`) are all removed from `compute-state.mjs`.
- **ADR-096's Announcement-closure branch retires** — there is no Announcement object for the FYI-closure rule to describe. ADR-096's *general* claim — every Discussion is closed by its initiator, and a Discussion opened to converge something needs a summarizing final comment first — is **not** Announcement-specific and survives unchanged, folded directly into `settle`'s contract (blueprint section 10: "Only the designated seat settles — a Discussion's host, an Issue's acceptor").
- **The awareness/ACK leg of ADR-091 and ADR-092 retires outside legacy compatibility.** Their three-way split (awareness / exact review / evidenced work) is subsumed by the blueprint's own three-way confusion-list (blueprint section 10's reply contract draws the same line natively: assignment vs. requested review vs. a stated completion rule) — the *review* leg (a PR verdict) and the *evidenced-task* leg (an assignee's stated evidence) fold unchanged into the new Issue/PR contracts; only the *awareness* leg (an ACK Announcement's `👀`) has no home in the four-object model and survives solely inside the LEGACY `[ACK]`-titled Discussion compatibility path described below.
- **ADR-093's Announcement-specific hard rule retires**: "every report tags a recipient, FYI included" presumed an Announcement object to tag. Its other two contributions are **not** superseded: the obligations/notices two-tier split itself, and the ratified invariant that an open non-draft PR is never obligation-free, both remain exactly as stated — PRs are untouched by this wave. The notices tier's *scope* changes shape: instead of narrowing to "wild traffic left over after receipt-default absorbed most mentions" (ADR-097's framing), it now catches essentially every plain `@mention` as the default landing spot, since there is no receipt-default to absorb any of them — this is the tier doing exactly its originally-stated job, not a new tier.

### What remains fully in force, unchanged

- **ADR-090** — the foundational premise (GitHub owns collaboration state; Relay owns routing, obligation semantics, verification) is deepened by the blueprint's two-memories model, never reversed.
- **ADR-098 (upstream nursery)** — its routing boundary, "can you state an owner and a completion rule yet?", *is* the blueprint's own section-1 boundary test, restated independently and arrived at the same place. Absorbed into the new router without change.
- **ADR-099 (decision reversal consent)** — its principle, that reversing an agreed decision needs a stated owner and completion rule rather than a passive awareness signal, is absorbed into the new model's stage-label mechanics (`awaiting-acceptance`/`awaiting-record` on an Issue) rather than the old "Decision Issue" shape specifically; the underlying consent requirement is unchanged.
- **ADR-095 (author sign-off)** — applies identically to every outbound-prose writing verb regardless of which object model is underneath; untouched by this wave.
- **ADR-094 (entry templates)** — the launch-audits-templates convention is unchanged; the specific template *set* is updated in this same wave (`announcements.yml` removed, `needs-input.yml` and `tell.yml` added) as a refinement, not a supersession — see Consequences.

### The new model, briefly

Four objects with clean boundaries (Discussion = think together, Issue = get one thing finished, PR = review a change, Commit = keep what's established); two memories (GitHub = collaboration memory, `decisions/`/`briefs/`/`core/` = formal memory); every closed Issue/Discussion passes a promotion test (does this need a new Decision file, or does the closing Resolution comment suffice); Decisions carry stable IDs and mechanical `status: active|superseded` supersession; Briefs and Core cite Decisions, never restate them; and a three-tier metadata law (native fields for responsibility, structured prose for semantics, a conformance sweep for health) keeps the digest reducer honest about what it can and cannot compute from typo-proof signals alone. Full detail: the blueprint, section by section.

### Legacy compatibility

A `[ACK]`-titled open Discussion keeps its **original pre-097 single-recipient** `👀` semantics (the first `@mention` in title/body is the designated recipient; later mentions own nothing) so a repository mid-migration does not silently lose an in-flight legacy obligation. This path is marked `LEGACY` in code and SKILL.md prose and is expected to retire once legacy `[ACK]` Discussions have been migrated off — no new `[ACK]`-titled Discussion should be created under the new model (see `report/SKILL.md`).

> **Retired 2026-07-22.** Migration completed with zero live `[ACK]`/`ack-required` Discussions remaining in the repo, so this compatibility path retired as planned. `isLegacyAckDiscussion`, `legacyAckRecipient`, and the single-recipient ACK obligation branch were removed from `compute-state.mjs`; `hasFormalSignal` no longer checks a legacy recipient. An `[ACK]`-titled Discussion is now just a Discussion — a plain `@mention` lands on the notices tier like any other. See `plugins/relay/CLAUDE.md`'s (now one-line) LEGACY note and `docs/design/relay.md`'s changelog for the full retirement record.

## Consequences

- **`plugins/relay/skills/digest/scripts/compute-state.mjs`** — `SCHEMA_VERSION` 3 → 4. Removed: `isAnnouncementDiscussion`, `announcementRecipients`, `authorizedResolution`, the `finalResolution`/`issueType` parsing in `normalizeLive`, the per-recipient ACK obligation block, and the `close-announcement` nudge. Added: `isLegacyAckDiscussion (since retired 2026-07-22, see the note above)`/`legacyAckRecipient` (LEGACY, single-recipient), `ISSUE_STAGES`/`issueStage` (native stage-label obligation derivation for Issues: `needs-input` → `provide-requested-input`, `awaiting-acceptance` → `accept-or-dispose`, `awaiting-record` → `record-decision`, none → the unchanged default `act`; conflicting labels are malformed and flagged `malformed: ['conflicting-stage-labels']`, resolved to the latest stage in canonical order). `hasFormalSignal` now checks the legacy recipient instead of the receipt-default recipient list. PR logic is untouched. `compute-state.test.mjs`: 76 → 81 tests, all green — receipt-default/close-nudge tests removed or rewritten onto the legacy path, stage-label tests added (one per action, conflicting labels, assignee-less Issue), every PR/Q&A/notice/temporal test kept unmodified.
- **`plugins/relay/skills/digest/SKILL.md`** — obligation contract rewritten around stage labels; the Announcement/receipt-default section replaced with a LEGACY-labeled compatibility note; notices-tier framing updated to reflect its widened default scope; schema section states `schemaVersion: 4` and the new `malformed` field.
- **`plugins/relay/skills/report/SKILL.md`** — router rewritten to Issue-default per blueprint section 1/10 (tell → Issue; a diff review → PR; a non-diff review → Issue; a needs-input Issue for an owned question, with the three form fields; Discussion reserved for genuine convergence; a born-crisp ask still opens the Issue directly); entry-owner discipline and the ADR-098/099 clauses carry over; the sign-off gate carries over.
- **`plugins/relay/skills/reply/SKILL.md`** — rewritten to the blueprint's reply contract: acts only on existing objects; the baton flip (`needs-input` ⇄ `awaiting-acceptance`, native label toggle plus reassignment to the acceptor) replaces the old ACK-centric response map; `👀` retires except inside the legacy `[ACK]` path; sign-off gate carries over.
- **`plugins/relay/skills/settle/SKILL.md`** — rewritten to the blueprint's settle contract: the settlement block (`Resolution:` / `Reason:` / `Decision required: Yes/No` + `Recorder:` / `Follow-ups:`), the native promotion signal chain (`awaiting-record` label plus reassignment to the recorder on Issues; recorder defaults to the settlement comment's author on Discussions), the Decision file format (section 3, including frontmatter and the supersession-marking exemption from the single-purpose direct-commit rule), and the derived-view citation law.
- **`plugins/relay/skills/brief/SKILL.md`** — the derived-view law: a Brief integrates only `active` Decisions, cites `[D-0xx]` anchors for every claim, never restates authoritative Decision wording, and is always a PR; staleness is defined as citing a `superseded` Decision.
- **`plugins/relay/skills/launch/SKILL.md`** and its `references/templates/` — `DISCUSSION_TEMPLATE/announcements.yml` deleted; `q-a.yml` kept; new Issue Forms `needs-input.yml` (Question / Done when / After reply) and `tell.yml` (What to take note of / Owner-to-assign note); the audit checklist gains the four stage/opt-out labels (`fyi`, `needs-input`, `awaiting-acceptance`, `awaiting-record`) and the `decisions/` scaffold (directory, README index note, frontmatter spec) as named findings.
- **`plugins/relay/CLAUDE.md`** and **`docs/design/relay.md`** — lifecycle matrix and router prose rewritten to the four-object/two-memory model, the Resolution/promotion test, verb contracts, the three-tier metadata law, and the indexes law (headers-as-index-rows, derived never hand-edited). `CLAUDE.md`'s refines list gains this ADR and a pointer to the blueprint.
- **`plugins/relay/skills/migrate/SKILL.md`** — untouched; still the explicit-only v0-to-v1 legacy bridge, orthogonal to this wave.
- **relay `1.4.0` → `2.0.0`** — the Announcement/receipt machinery's removal is a real behavior break for any repository that had adopted receipt-default traffic, so this is a major bump, not a minor one.
- No new daily verb. The daily verb set stays `launch · report · digest · reply · brief · settle` plus explicit-only `migrate`.

## Out of scope

- The counterpart's coordination repository's own adoption of `decisions/` + the frontmatter spec — that lands as a separate PR the counterpart reviews (their consent is the gate, per ADR-099's still-standing principle).
- Live objects predating this model: they finish under the rules they were opened under, then close; nothing is migrated retroactively (blueprint section 11).
- A conformance-sweep tool that mechanically checks the three-tier metadata law's tier-3 health signals (`Canonical record:` resolves, `Follow-ups:` opened, every closed object carries a `Resolution:`) — named as future work by the blueprint, not built in this wave.
