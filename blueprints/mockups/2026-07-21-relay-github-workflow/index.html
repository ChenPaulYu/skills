資訊怎麼從證據變成當前真相？

三層不是必經漏斗；點擊層級查看內容，再比較下面兩條各自可選的路線。

路線 1 · 可選的綜合判讀
一個或多個 Raw GitHub objects→brief→briefs/*.md + citations
不是每件事都要走。只在理解要跨情境重用、且不適合只留在單一 object 的 final resolution 時使用；不關閉 object、不代表共識，也不修改 Core。
路線 2 · 有權限的當前契約
Decision Issue 或精確 diff→Core PR + approval→core/
brief 不是 Core 的前置階段。Core PR 可以直接引用 Decision Issue；merge 才生效。

最小實作長什麼樣？

Relay 是 GitHub primitives 上的一層薄協定，不另蓋一座狀態倉庫。

一個 workspace = 一個 repo

v1 不跨 repo 聚合；repository 本身就是協作與權限邊界。

沒有平行資料層

不建立 DB、frontmatter、thoughts/ 或 logs/；直接讀寫 GitHub objects、reactions、reviews 與 timeline。

brief 只是普通 Markdown

一份 .md + citations + PR；briefs/README.md 擁有導覽與自組織規則。

失敗後沿 URL resume

partial failure 回報已建立的 Discussion／Issue／PR URL；下一次從該 object 繼續，不重建、不複製。

六個 skills，六種責任

點擊 skill，看看它負責什麼，以及哪些狀態刻意交給 GitHub。

brief 為何獨立？generic doc authoring 能寫文件，但不擁有 currentness、citations、briefs/README.md 與 retirement。
reply = 留下我的回應；settle = 用權限宣告整件事結束或生效。
v1 · GitHub-native——一個 Relay workspace 對應一個 GitHub repo；沒有平行 DB 或 frontmatter。 partial failure 保留既有 URL，resume 而不重建。
Mention ≠ 有義務只有 assignment、review request 或明確的 ACK 標記，才會產生待辦。
Comment ≠ 已核准目前 revision 收到必要 verdict 之前，review 都還沒完成。
Closed ≠ 已決定Decision Issue 關閉前，必須有具權限者留下 final resolution。
Core = 現在為真未來才生效的決策先留在 Core 外；Core PR merge 才正式生效。
