# Relay roster initialization and recipient resolution — plan

> Generated: 2026-07-23 · Spec source: Ricercar-to-Accord Relay session and `docs/observations/2026-07-23-relay-must-resolve-recipients-before-writing.md` · Stage 1: fresh

## Context

Relay 2.1.0 defines `relay.yml` as PR-attested reference data in `plugins/relay/CLAUDE.md` and ADR-101, but no active skill creates it. `launch` explicitly treats an absent roster as `missing (optional)` and forbids including it in the mutation plan; `migrate` can preserve an existing roster but is not a setup path. Accord previously carried a valid roster, but the GitHub-native migration removed it before ADR-101 restored the roster contract without restoring an initialization owner.

At the same time, `report` requires a concrete assignee or reviewer but only says to resolve the current repository and distill an owner. It does not define how a human-facing name or role becomes a verified GitHub account, nor how the Accord repository is selected when the current working directory is a project repository. The old file-based Relay had a real resolver (`$RELAY_REPO` → cached path → marked cwd → ask), but ADR-090 removed it when Relay became GitHub-native and declared the current repository to be the workspace. That removed the only bridge from a sibling project repo to the central Accord repo. In the prompting session this double gap caused two guessed handles and an Issue created in the wrong repository before the user corrected both destination and recipient.

The source skills under `plugins/relay/` are the editable owners. `.agents/skills/relay-*` and root `AGENTS.md` are generated by `scripts/build-codex.mjs`; they must be regenerated, not edited directly. Because this adds behavior to two existing skill doors and materially changes Relay's advertised setup/report contract, it also requires a minor version bump and synchronized human-facing surfaces.

Intent in one sentence: **make Relay initialize an attested identity roster through a reviewed PR, then require `report` to resolve both destination and recipient from verified facts before it creates an owned GitHub object.**

## Resolved questions

| Question | User's answer |
|---|---|
| How is a missing roster initialized? | `launch` gathers roster data, previews the exact YAML and PR mutations, then creates a branch and PR after approval; it never writes directly to the default branch. |
| What schema does the roster use? | `people.<legacy_handle>.{name, github, title, git}`; `name`, `github`, and `title` are required, while `git` is optional and never blocks recipient routing. |
| May `report` use an explicitly supplied GitHub account? | Yes. It verifies the account and may proceed without a roster match. Inferred handles are forbidden. |
| What happens when identity is missing or ambiguous? | Stop before preview/creation and ask for the GitHub account; never guess from display-name similarity, organization names, or previously seen handles. |
| How does an agent find Accord from a sibling project repository? | Resolve in order: an explicit object/repo, `$RELAY_REPO`, `~/.config/relay/repo`, a cwd that can prove it is the Relay workspace, then ask. `launch` may preview and persist the verified `OWNER/REPO` as the machine's default pointer. |
| Is the default-repo pointer collaboration state? | No. It is local routing configuration containing only one verified `OWNER/REPO`; GitHub remains the owner of collaboration state and `relay.yml` remains the owner of attested identity data. |
| Which verbs change? | Only `launch` and `report`, plus their shared Relay contract, ADR-101, projections, release metadata, surfaces, observation, and verification. Other Relay verbs stay unchanged. |

## Approach

### Step 1 — make the roster contract complete and single-owned

1. Amend `plugins/relay/CLAUDE.md` so the attested-reference section owns the canonical `relay.yml` schema, required/optional fields, matching rules, and the distinction between identity reference data and GitHub-native obligation state.
2. Define the default Relay-repository resolver shared by `launch` and `report`:
   - an explicit object URL or `OWNER/REPO` supplied for the current action wins;
   - otherwise use `$RELAY_REPO` when set;
   - otherwise read one verified `OWNER/REPO` from `~/.config/relay/repo`;
   - otherwise accept cwd only when repository-owned Relay markers prove that cwd is itself the intended workspace;
   - otherwise ask the user and offer to persist the verified answer;
   - validate every candidate with `gh repo view` before use, and self-heal a stale local pointer by falling through rather than targeting a guessed replacement.
3. Define the identity preflight shared by roster-aware writes:
   - accept a user-supplied GitHub username only after `gh api users/<handle>` verifies it;
   - otherwise match a human-facing legacy handle, name, or role against `relay.yml`;
   - treat zero or multiple matches as unresolved and ask the user;
   - never guess a GitHub account.
4. State explicitly that `~/.config/relay/repo` is local routing configuration, not formal memory or a parallel collaboration-state store. Amend ADR-101 to record both missing initialization owners: `launch` may persist the verified default repo locally after approval, and may create or update the roster only through a counterpart-reviewed PR. Keep `digest` independent of both files.

### Step 2 — give `launch` the roster-initialization path

1. Change `plugins/relay/skills/launch/SKILL.md` so an absent roster is a `missing (optional)` finding with an available, separately gated remedy rather than a forbidden mutation.
2. After resolving and verifying the repository, include an optional local mutation in the preview: write its canonical `OWNER/REPO` to `~/.config/relay/repo` as this machine's default Relay destination. Read the pointer back after writing it. This does not require counterpart review because it changes no repository or shared fact.
3. Require `launch` to gather and preview the complete proposed roster before any repository write. Each row uses the canonical schema; `git` may be omitted.
4. Validate the proposed mapping before opening a PR:
   - YAML shape is parseable;
   - legacy handles are unique;
   - required fields are present;
   - every `github` account resolves through GitHub;
   - the named counterpart/reviewer is one of the verified individual accounts and is not the PR author.
5. After author approval, create a normal branch and PR in the resolved Relay repository, request the counterpart as reviewer, and read back the file diff, PR URL, head revision, and requested reviewer. The roster becomes attested only after counterpart approval and merge; `launch` must report a pending PR honestly until then.
6. Apply the same PR-only path to later roster edits; direct default-branch writes remain forbidden.

### Step 3 — make `report` resolve destination and recipient before author sign-off

1. Change `plugins/relay/skills/report/SKILL.md` process step 1 into a combined destination-and-recipient preflight.
2. Resolve an explicitly supplied object URL first. Otherwise run the shared resolver (`explicit repo → $RELAY_REPO → ~/.config/relay/repo → proven Relay cwd → ask`); do not silently use the current project repository merely because it has a Git remote.
3. Resolve the recipient using the shared contract:
   - explicit verified GitHub username wins;
   - otherwise use an unambiguous roster match;
   - missing roster, zero matches, or multiple matches require a user question before the preview;
   - no guessed fallback.
4. Include the resolved `OWNER/REPO`, object type, owner/reviewer, and every follow-up mutation in the existing author-signoff preview.
5. Keep the existing create, assignment/review request, read-back, and partial-failure rules unchanged after preflight succeeds.

### Step 4 — release and project the behavior

1. Bump Relay `2.1.0 -> 2.2.0` in `plugins/relay/.claude-plugin/plugin.json`; this is a backward-compatible new capability on existing verbs.
2. Update the hand-owned Relay marketplace description, `README.md`, and `docs/site/index.html` so they say:
   - `launch` can remember the verified default Relay repository locally and initialize/update the attested roster through a reviewed PR;
   - `report` resolves the default Accord destination and verifies the recipient before creating owned traffic;
   - absence or ambiguity prompts the user instead of triggering a guess.
3. Mark `docs/observations/2026-07-23-relay-must-resolve-recipients-before-writing.md` as promoted and link the landed contract/skill targets.
4. Run `node scripts/build-manifests.mjs` and `node scripts/build-codex.mjs` to regenerate Cursor, marketplace version, Codex skills, and root guidance from their owners.

### Step 5 — verify the contract and the real workflow

1. Run the repository validators and Relay tests.
2. Inspect the generated `relay-launch` and `relay-report` copies to confirm the source behavior survived Codex lowering intact.
3. Run a dry behavioral walkthrough against these cases without creating throwaway live objects:
   - a sibling project repo resolves to the configured Accord `OWNER/REPO` rather than its own remote;
   - a stale local pointer falls through to an explicit question and never silently selects cwd;
   - missing `relay.yml` produces an optional missing finding plus a PR remedy;
   - explicit verified username can proceed;
   - human-facing name resolves through one roster row;
   - absent roster, no match, and duplicate match all stop before author sign-off;
   - current project repository is not mistaken for Accord when Accord is the intended destination.
4. Run `launch` read-only against the Accord repository and confirm it now recommends roster initialization instead of claiming no skill owns it. Do not open the live roster PR as part of the skills-repo implementation unless the user separately approves that outward mutation.

## Out of scope

- Creating Accord's live `relay.yml` PR; that is a separate outward mutation requiring its own preview and approval.
- Making `digest` consume roster data; obligations remain derived only from native GitHub fields.
- Adding recipient logic to `reply`, `brief`, `settle`, or `migrate` unless later evidence shows those verbs independently translate human names into new native ownership.
- Building a new executable roster parser or Relay database; this change is a skill contract and workflow correction.
