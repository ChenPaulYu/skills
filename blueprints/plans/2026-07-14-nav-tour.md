# `nav:tour` — implementation plan

> Generated: 2026-07-14 · Spec source: this session's `tour` naming decision · Stage 1: fresh grounding against the current nav roster, `nav:map`, `reflect:catchup`, the prior generic-teach ruling, and marketplace registration gates
>
> Status: plan only. This artifact changes no skill, manifest, generated mirror, README, or site-map surface.

## Context

The requested job is narrower and more disciplined than a generic `teach`: guide a user through what the **current codebase** can do, how its load-bearing parts cooperate, and which recorded decisions explain its present shape; then expose the difference between the agent's model and the user's model. The deliverable is not a lecture. It is a corrected **shared system model**.

The closest existing door is `nav:map`. It already calls the codebase map a teaching projection and its current frontmatter explicitly fires on onboarding language, but its mandatory deliverable is a durable bilingual HTML render at `docs/codebase-map/index.html`. That trigger is a real collision to resolve during implementation: `map` must narrow to requests that ask to create/refresh/open the durable visual map; conversational onboarding moves to `tour`. `map` is periodic, mechanical-tier, and write-producing. `tour` has a different cadence and medium: summoned when a person wants to understand or re-align, judgment-heavy, conversational, read-only, and complete only after a correction loop. This is the same kind of cadence/output split that justified `sync` versus `map` in ADR-029.

The prior ruling in `blueprints/thoughts/2026-07-14-elicit-ecosystem-four-quadrants.md` rejected a generic `teach` because "inject context" alone is a thin wrapper around default explanation, `frame:analogize`, or external research. `tour` clears that objection only if the **model-delta loop is mandatory**: ground a system model, expose it for correction, classify the correction, and return the revised shared model. If implementation drops that loop and merely summarizes the repo, the skill must not ship.

`tour` belongs in nav because its object is the already-existing codebase. It may read decision artifacts owned elsewhere, but it never authors or changes decisions. `shape` continues to own convergence about what should be built; `tour` explains what exists and what recorded choices shaped it.

## Resolved questions

| Question | Decision |
|---|---|
| Name | `tour` — action-oriented and specific to a guided walkthrough; not generic `teach` or ambiguous `orient` |
| Plugin | `nav` — the object is an existing codebase and its navigability |

## Skill contract

### One-line purpose

Guide the user through what a current codebase does, how it works, and why recorded decisions shaped it; then reconcile the agent's model with the user's corrections.

### Trigger surface

The frontmatter description should lead with the verb and cover these intents without synonym stacking:

- "walk me through this codebase" / "帶我逛這個 repo"
- "what features does this system have?"
- "explain how this repo works and why it was designed this way"
- "onboard me to this codebase"
- "align our understanding of the current system"

Recommended frontmatter direction, to refine during implementation:

```yaml
name: tour
description: "Guide a user through what the current codebase does, how its main parts cooperate, and which recorded decisions shaped it; then reconcile the agent's model with the user's corrections. Use for conversational codebase walkthroughs, onboarding, or shared-understanding checks. Read-only and in-chat; not a map refresh, work-status catchup, or pre-decision survey."
```

Keep the final description within the marketplace and Codex description budgets. Add a separate trigger-first `nav-tour` projection to `platforms/codex/descriptions.json`; do not assume the Claude description is copied unchanged.

### Fixed two-turn protocol

#### Step 1 — Scope the tour

- Read the user's stated focus first: whole repo, subsystem, user journey, or one architectural question.
- If audience/depth materially changes the answer and is absent, ask one concise question; otherwise choose a progressive top-down tour and state the assumed depth.
- Never turn the opening into an interview. One high-impact clarification is the maximum before grounding.

#### Step 2 — Ground from durable evidence

Consume repo knowledge in a tolerant three-tier order and self-report the tier:

1. **Standard navigation layer** — current `docs/codebase-map/index.html`, load-bearing file headers, README/architecture docs, ADRs or decision records.
2. **Non-standard/ad-hoc documentation** — whatever readable overview, design notes, changelog, tests, or decision prose the repo actually has.
3. **Source-and-git fallback** — entry points, public interfaces, routes/commands, tests, imports, configuration, and relevant commit bodies.

The map is a starting index, not unquestioned truth. Revalidate load-bearing claims against current source or tests when the map/doc may be stale. Absence of the canonical map is degradation, not failure; `tour` must still work and report that it used the fallback tier.

Classify every `why` claim by provenance:

- **Recorded** — supported by an ADR, decision doc, commit body, or explicit user statement.
- **Inferred** — architecture/code suggests it, but no durable rationale was found.
- **Unknown** — neither evidence nor a defensible inference exists.

Code proves what exists and often how it works; code alone does not prove why a decision was made. Never upgrade an inference into a historical fact.

#### Step 3 — Deliver the guided model

Walk top-down in this fixed order, progressively disclosing detail:

1. **System in one sentence** — who it serves and what outcome it produces.
2. **What it can do** — user-visible capabilities or externally observable behaviors, grouped by user journey rather than folders.
3. **How it works** — the smallest useful end-to-end flow through the load-bearing domains; name interfaces before internals.
4. **Why it has this shape** — only load-bearing decisions, each marked Recorded / Inferred / Unknown and linked to its durable evidence when available.
5. **Constraints worth carrying** — boundaries a future change must not accidentally violate.

Use plain Traditional Chinese for user-facing explanation, preserve code identifiers where useful, and use at most one deliberately chosen analogy for the hardest structural point. `tour` does not run `frame:analogize` or compare multiple analogy candidates.

#### Step 4 — Expose the proposed shared model

End the first response with 4-7 short, falsifiable statements under **"我目前認為我們共同理解的是"**. They must cover capability, flow, a load-bearing decision, and a constraint. Ask one correction prompt:

> 哪一句不對、少了重要前提，或只反映 code 現況但不是你的原意？

Do not ask the user to restate the whole system. The agent puts its model on the table; the user edits it.

#### Step 5 — Land the alignment delta

On the user's reply, classify each change:

- **Confirmed** — both models already matched.
- **Corrected** — the agent's factual or historical model was wrong.
- **Missing** — the agent omitted a capability, decision, or constraint.
- **Intent/code divergence** — the user's intended system differs from shipped reality.

Return only:

1. **認知差集** — the changes, with evidence or uncertainty.
2. **更新後共同模型** — the corrected 4-7 statements.
3. **Unresolved** — facts still requiring source inspection or a user decision.

If an intent/code divergence appears, report it without silently editing code or decision artifacts. Offer the correct next door only when useful: `shape-elicit` for a new decision, `nav-plan` for a decided code change, or `nav-map` when the durable map itself is stale. Offers, never calls.

### Completion criterion

A `tour` invocation is complete only when:

- capability, system flow, recorded rationale, and constraints are grounded separately;
- every rationale claim is provenance-labeled;
- the proposed shared model is stated as falsifiable claims;
- the user is given the correction door;
- after a correction reply, the alignment delta and revised shared model are returned;
- no source, plan, map, ADR, or decision artifact was modified.

If the user ends after the first response, report that the tour is grounded but alignment remains unconfirmed; do not pretend the shared model was validated.

## Boundaries

| Neighbor | Boundary |
|---|---|
| `nav-map` | `map` writes/refreshes the durable bilingual repo projection; `tour` consumes available evidence and conducts an in-chat correction loop. Map answers "where is everything?"; tour answers "what model should we now share?" |
| `reflect-catchup` | `catchup` reconstructs current work state: goal/done/now/open/next. `tour` explains the relatively stable product/system model, not where today's task stopped |
| `shape-survey` | `survey` maps missing axes before a decision and reports the gap in the user's decision-space knowledge. `tour` explains the system that already exists and checks factual/shared understanding |
| `shape-elicit` | `elicit` converges a new principle or decision with the user. `tour` may expose an intent/code divergence but cannot adjudicate it |
| `frame-analogize` | `analogize` builds and stress-tests an analogy for one already-understood concept. `tour` builds a grounded whole-system model; one explanatory analogy is style, not its engine |
| `nav-audit` | `audit` assesses architectural health and reports findings. `tour` teaches the current system without grading or fixing it; smells noticed incidentally are marked, not investigated into an audit |
| README/onboarding docs | Static docs publish one durable explanation. `tour` is live, evidence-checked, audience-sensitive, and requires a correction loop; it does not replace those docs |

## Anti-patterns the SKILL.md must refuse

| Temptation | Instead — and the tell |
|---|---|
| Dump a folder-by-folder summary | Group capabilities by user journey and architecture by load-bearing flow. Tell: the response mirrors `find` output rather than explaining behavior |
| Treat a codebase map as current truth | Revalidate load-bearing claims and self-report the evidence tier. Tell: citing an HTML blurb without checking source/test reality |
| Invent historical rationale from code shape | Mark it Inferred or Unknown. Tell: using "we chose X because" with no decision evidence |
| Finish after the lecture | Put forward falsifiable shared-model statements and request correction. Tell: the response ends with "hope that helps" rather than an alignment door |
| Ask the user to teach the repo back from scratch | Offer the agent's model for editing. Tell: broad questions like "what do you think this system does?" after already reading the repo |
| Turn a disagreement into an implementation | Classify intent/code divergence and offer the right next verb. Tell: editing code or plans during a read-only tour |
| Reproduce the whole map in chat | Select the smallest model that explains capabilities, flow, decisions, and constraints. Tell: every domain and file receives equal airtime |

## Approach

### Phase 1 — Author the skill and its rationale

1. Create `plugins/nav/skills/tour/SKILL.md` as a self-contained, read-only skill implementing the contract above.
2. Keep it under 300 lines if possible; no bundled reference is expected because the engine is compact. Add a reference only if implementation would otherwise duplicate a long evidence rubric.
3. Restate nav's eight rules verbatim, applying them to the tour where useful; do not edit the rules themselves.
4. Add ADR-082 under `docs/adr/` covering:
   - evidence that the generic `teach` ruling remains valid;
   - why mandatory shared-model delta makes `tour` non-No-Op;
   - why it belongs in nav rather than shape;
   - why it is separate from `map` despite consuming it (medium, cadence, write behavior, judgment tier);
   - the seven neighbor boundaries above;
   - re-evaluation condition: retire/merge it if real use shows users only consume the first-pass explanation and never use the correction loop.
5. Narrow `plugins/nav/skills/map/SKILL.md` frontmatter only where needed to remove conversational onboarding as a `map` trigger. Preserve map's body, teaching-projection role, render workflow, and write contract; requests to create/refresh the durable HTML still belong to `map`.

### Phase 3 — Validate routing and behavior before calling it shipped

#### Routing probe

Create a findings artifact under `docs/findings/` and test at least 15 prompts:

- **Positive tour triggers (5+)** — whole-repo walkthrough, feature explanation, architecture-and-decisions explanation, conversational onboarding, shared-model alignment.
- **Boundary negatives (6+)** — map refresh → `nav-map`; work cursor → `reflect-catchup`; pre-decision blind spots → `shape-survey`; decide a new principle → `shape-elicit`; explain one concept differently → `frame-analogize`; architecture health review → `nav-audit`.
- **Focused variants (4+)** — one subsystem, one user journey, repo without map, repo with map but no decision records.

Pass criteria: all clear positives select `tour`; all boundary negatives stay with their owner; ambiguous prompts may ask one scope question but must not silently route to a wrong deliverable.

#### Behavioral probe

Run the skill against at least two real repositories:

1. **Well-documented repo** — map/headers/ADRs exist. Verify Recorded rationale cites evidence and the first response ends with falsifiable shared-model statements.
2. **Sparse or non-standard repo** — no canonical map and incomplete decisions. Verify tolerant fallback, tier self-reporting, and honest Inferred/Unknown labels.

For at least one run, provide a deliberate user correction and verify the second response distinguishes Confirmed / Corrected / Missing / Intent-code divergence without modifying files.

Record failures and iterate the SKILL.md before release. A prose inspection alone is insufficient because the correction loop is the feature's moat.

## Out of scope

- Implementing the skill in this planning session.
- Refreshing a target repo's codebase map or writing onboarding documentation during a tour.
- Teaching arbitrary external concepts; that remains normal explanation, `frame-analogize`, or research depending on the requested deliverable.
- Deciding product direction or reconciling intent/code divergence; `tour` only exposes the divergence.
- Automatically persisting the shared model. A future need for durable user-specific onboarding notes requires separate evidence and ownership analysis.
