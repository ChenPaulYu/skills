# observe — harvest this session into a durable observation

Capture a **durable, reusable learning** from the current session and write it into a **knowledge base** (`docs/observations/`). This is the first step of a repo-evolution loop: **lived experience → observation → ADR → skill change.** Where `manage-summarize` is the complete objective recap, observe is **selective** — it keeps the *one* insight worth keeping.

Optional focus from the user: **$ARGUMENTS** (if empty, scan the whole session for the most valuable learning).

## Step 1 — Locate the knowledge base

```bash
TARGET="${SKILLS_REPO:-.}"            # central repo if set, else the current project
ls "$TARGET/docs/observations" >/dev/null 2>&1 && echo "ok: $TARGET/docs/observations" || echo "no observations dir at $TARGET/docs/observations"
date +%F
```

- Prefer `$SKILLS_REPO` (a central knowledge base); else fall back to the **current project's** `docs/observations/`.
- If neither exists, ask the user where observations should live (or whether to create `docs/observations/` in the current project) — don't write to a guessed path.
- Use the real `date +%F` output for the filename + frontmatter — never guess the date.

## Step 2 — Distill the learning (the hard part — not a transcript)

Reflect on THIS session and extract the **non-obvious, reusable mechanism or principle** — what future-you would want to know, not a play-by-play. A good observation names a *signal* and a *move*, grounded in what actually happened. Ask:

- What non-obvious insight / mechanism / failure mode did this session surface?
- What's the reusable principle, stated as a **claim** (the title is a full sentence)?
- What concretely is the evidence (this session) — files, decisions, what was tried?

One observation per run — don't dump several. If `$ARGUMENTS` is given, anchor on that; else pick the single most valuable learning. (If `manage-summarize` ran earlier this session, reuse its recap as the raw material to distill from.)

## Step 3 — Dedupe against existing observations

```bash
ls "$TARGET/docs/observations/"
```

If the learning **overlaps an existing observation**, prefer **appending/strengthening that file** (add a new evidence case, move it toward `landed` per the evidence gate) over creating a near-duplicate. Surface the overlap and let the user pick new-vs-append.

## Step 4 — Write it in the established format

Path: `"$TARGET/docs/observations/<YYYY-MM-DD>-<kebab-topic>.md"`. Read one existing file first to mirror its shape:

```markdown
---
date: <YYYY-MM-DD>
status: raw
---

# <the learning as a full-sentence claim>

> Source: <one line — this session, what it was doing>   (optional)

## What prompted it
<the situation that surfaced it>

## The signal
<the non-obvious mechanism / principle — the core>

## Evidence so far
- **Only case (<date>, <context>)**: <what grounds it>.

(One case → stays `raw`. <the trip-wire / what would promote it>.)
```

Conventions:
- **Progressive disclosure**: title is a claim; sections lead with their point; `head -12` yields the gist.
- **Link** related ADRs and observations relative to the knowledge-base root (e.g. `docs/adr/0xx-...md`, `[[observation-slug]]`); no `./` or `../`.
- **status: raw** for a single case (promotion is by *evidence*, not session count). Add a trip-wire line for what would graduate it.

## Step 5 — Flag, then hand back (don't over-reach)

- Note whether the observation likely **feeds an existing ADR** (name it) or is a **candidate for a future ADR** — but **do NOT write the ADR**. Observations are raw evidence; ADRs come later, deliberately.
- **Do not** edit any code, `SKILL.md`, manifest, or site map — observe only adds the observation.
- Show the written file's path + contents (or the diff if you appended); let the user review/commit. Don't commit unless asked.
